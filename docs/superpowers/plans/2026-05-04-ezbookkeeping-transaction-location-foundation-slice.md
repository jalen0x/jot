# ezBookkeeping Transaction Location Foundation Slice Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add optional latitude/longitude storage for transactions and expose it through the existing HTML create flow and JSON create/list API.

**Architecture:** Store coordinates directly on `transactions` as optional decimal columns with database range and pair constraints. `TransactionRecorder` remains the only transaction creation seam and accepts both Rails form fields (`geo_latitude`, `geo_longitude`) and API-style nested input (`geo_location: { latitude:, longitude: }`). This slice intentionally does not add map tiles, reverse geocoding, browser geolocation prompts, map previews, location editing, or location filters.

**Tech Stack:** Rails 8.1, PostgreSQL check constraints via `structure.sql`, Minitest model/service/integration/API tests, existing Flowbite semantic form classes.

## Logical Model

- `Transaction#geo_latitude`: optional decimal latitude, valid only from `-90` through `90`.
- `Transaction#geo_longitude`: optional decimal longitude, valid only from `-180` through `180`.
- A location is present only when both values are present. The database rejects half-locations.
- Coordinates belong to the existing user-owned transaction; no separate `TransactionLocation` entity is needed until maps need location metadata.

## Regression Risks Covered

- HTML transaction creation does not drop user-entered coordinates.
- API clients can send legacy-shaped `geo_location` and receive it back in response JSON.
- External writers cannot persist out-of-range coordinates or half-locations by bypassing Rails validations.
- Existing transactions without coordinates remain valid.

Create `db/migrate/20260504090000_add_geo_location_to_transactions.rb`:

```ruby
class AddGeoLocationToTransactions < ActiveRecord::Migration[8.1]
  def change
    add_column :transactions, :geo_latitude, :decimal, precision: 10, scale: 7, null: true, comment: "Optional geographic latitude"
    add_column :transactions, :geo_longitude, :decimal, precision: 10, scale: 7, null: true, comment: "Optional geographic longitude"

    add_check_constraint :transactions,
      "geo_latitude IS NULL OR geo_latitude BETWEEN -90 AND 90",
      name: "transactions_geo_latitude_range"
    add_check_constraint :transactions,
      "geo_longitude IS NULL OR geo_longitude BETWEEN -180 AND 180",
      name: "transactions_geo_longitude_range"
    add_check_constraint :transactions,
      "(geo_latitude IS NULL AND geo_longitude IS NULL) OR (geo_latitude IS NOT NULL AND geo_longitude IS NOT NULL)",
      name: "transactions_geo_location_pair"
  end
end
```

private

def geo_location_pair
  if geo_latitude.present? && geo_longitude.blank?
    errors.add(:geo_longitude, "can't be blank when latitude is present")
  elsif geo_longitude.present? && geo_latitude.blank?
    errors.add(:geo_latitude, "can't be blank when longitude is present")
  end
end

def geo_location_json
  return if geo_latitude.blank? && geo_longitude.blank?

  {
    latitude: geo_latitude.to_s,
    longitude: geo_longitude.to_s
  }
end
```

def coordinate_value(attributes, nested_key, direct_key)
  direct_value = attributes[direct_key]
  return direct_value if direct_value.present?

  geo_location = attributes[:geo_location]
  geo_location = geo_location.to_h.symbolize_keys if geo_location.respond_to?(:to_h)
  geo_location[nested_key] if geo_location.present?
end
```
